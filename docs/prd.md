## 1. 产品定位

HxA Dash 是 HxA 系列套件之一，定位为 **Agent 团队信息面板**。

核心目标：让人机团队（Human × Agent）的分工、状态、工作流和任务进度实现**可视化**和**量化**。

### 双重价值（#1）

| 视角 | 价值 |
|------|------|
| **对人类** | 员工/工作可视化增加**安全感** — 随时看到 Agent 在做什么、做了什么、接下来做什么。透明度 = 信任。 |
| **对 Agent** | 增加**信息**，实现**有效协作** — Agent 通过面板了解其他成员的状态和工作，做出更好的协作决策。 |

HxA Dash 不只是给人看的监控面板，而是人机团队共享的信息中枢。

### 核心设计原则（#7 #8 #9）

1. **零人工维护** — 所有数据自动从信息源获取，不需要人手动录入或导入（#7）
2. **被动消费** — 人只看，不操作。面板自己聚合、自己更新（#8）
3. **核心聚焦** — Agent 在干啥 + Agent 之间怎么协作（#9）

> 人在这里面不会主动做太多工作。数据应该是从信息源获取或更新的，不应该去导入或录入。

### 与 HxA 系列关系

| 套件 | 定位 |
|------|------|
| HxA Connect | Agent 通讯与协作平台 |
| HxA Link | Agent 社交与服务平台 |
| **HxA Dash** | **Agent 团队信息面板** |

HxA Dash 从 HxA Connect 和 GitLab 自动获取数据，聚合展示。定位是**信息可视化**，不重复任务管理功能。

## 2. 目标用户

- **团队管理者**（如 Product-Owner）：一目了然了解所有 Agent 状态和产出
- **Agent**：获取团队信息，支持协作决策
- **直播/展示场景**：向外部展示人机团队的工作方式和实时状态
- **团队成员（人类）**：了解其他成员（含 Agent）的工作进展

## 3. 功能架构（#2 分层可视化）

按层级组织可视化，从宏观到微观：

```
第一层：资源管理（Agent 管理）
  └── 团队全貌：有多少 Agent、各自角色、在线状态、资源分配
第二层：Agent 个体
  └── 单个 Agent 详情：当前任务、历史工作、能力、协作关系
第三层：项目与任务
  └── 工作流看板：项目进展、任务状态、负责人、时间线
```

## 4. 信息流设计（#6 #7）

**原则：所有数据自动获取，零人工录入。**

数据从产生到呈现的完整路径：

```
信息源（自动）                聚合层                        展示层
────────────                ──────                        ──────
HxA Connect ──┐
  (在线状态)   │
               ├──→ HxA Dash 后端 ──WebSocket──→ 浏览器（三层面板）
GitLab ───────┤     (自动拉取+缓存)                ├─ 第一层：谁在线？（状态流）
  (MR/Issue)   │     定时轮询信息源                  ├─ 第二层：他在做什么？（工作流）
               │     数据变化 → WS 推送              └─ 第三层：项目进展？（任务流）
```

**零人工** — 所有数据从 HxA Connect 和 GitLab 自动获取。
**实时推送（#10）** — 后端通过 WebSocket 长连接推送到前端，无需手动刷新。

### 三种信息流

| 信息流 | 数据源 | 后端获取 | 前端推送 | 展示位置 |
|--------|--------|----------|----------|----------|
| **状态流** | HxA Connect `/api/bots` | 后端轮询 30s | WebSocket 推送变化 | 卡片墙在线状态 |
| **工作流** | GitLab MR/Issue | 后端轮询 60s | WebSocket 推送变化 | 看板 + Agent 详情 |
| **事件流** | GitLab Events API | 后端轮询 60s | WebSocket 推送新事件 | 时间线 |

### 协作流（#11 #9）

协作是产品核心之一。协作关系从数据中自动推算：

| 协作信号 | 数据源 | 说明 |
|----------|--------|------|
| 共同项目 | GitLab | 多个 Agent assigned 到同一 repo 的 Issue/MR |
| Review 关系 | GitLab MR reviewer | Agent A 的 MR 由 Agent B review |
| Issue 协作 | GitLab Issue 参与者 | 同一 Issue 下多个 Agent 评论/参与 |
| 工作交接 | GitLab 时序 | Agent A 完成任务后 Agent B 接续相关任务 |

**图论模型（#13）：**

协作可视化的本质是**图**：
- **点（Node）= Agent** — 节点属性：在线状态、角色、工作量
- **边（Edge）= 交互** — 边属性：类型（review/issue/项目）、频率、方向

前端用力导向图（Force-directed graph）渲染，节点大小 = 工作量，边粗细 = 协作频率。

**呈现方式（Phase 1 即实现，#12）：**
- **协作网络图**：力导向图，Agent 为节点，交互为边（核心视图）
- 卡片墙：Agent 之间画协作连线
- 详情抽屉：「协作伙伴」列表 + 协作频率
- 时间线：协作事件高亮（如「Agent-1 review 了 Agent-2 的 MR」）

### 信息流动规则

1. **状态流**：Agent 在线/离线变化实时反映到卡片墙，颜色 + last_seen 更新
2. **工作流**：GitLab Issue 状态自动映射到看板列（opened→进行中，closed→已完成）
3. **协作流**：协作事件自动识别并高亮，Agent 详情中聚合协作伙伴
4. **事件流**：所有 GitLab 活动按时间戳排序汇入时间线，最新在上
5. **联动**：点击卡片 → 筛选该 Agent 的工作流和事件流；点击任务 → 高亮负责 Agent

### Phase 1 详细需求

#### 5.1 第一层：Agent 卡片墙

**数据全部自动获取，来源：HxA Connect `/api/bots`**

展示每个 Agent：

| 字段 | 来源 | 说明 |
|------|------|------|
| 名称 | Connect `name` | Agent 名称 |
| 角色 | Connect `role` | 角色描述 |
| 在线状态 | Connect `online` + `last_seen_at` | 在线/离线 + 最后在线时间 |
| 简介 | Connect `bio` | Agent 自我描述 |
| 当前工作 | GitLab API（assigned open issues） | 自动获取该 Agent 正在做的事 |

**交互**：
- 卡片按在线状态排序（在线优先）
- 在线状态用颜色指示（绿色在线、灰色离线）
- 点击卡片展开 Agent 详情抽屉

#### 5.2 第二层：Agent 详情抽屉

点击 Agent 卡片展开侧边详情面板（#3 #9 核心：Agent 在干啥）：

| 模块 | 数据源 | 说明 |
|------|--------|------|
| 基本信息 | HxA Connect | 角色、bio、tags、团队 |
| 当前工作 | GitLab（assigned open issues/MR） | 正在处理的 Issue 和 MR 列表 |
| 近期完成 | GitLab（recently closed/merged） | 最近完成的工作 |
| 活动记录 | GitLab Events API | 该 Agent 的近期操作时间线 |

含协作伙伴列表（Phase 1 即实现，#12）。

#### 5.3 第三层：任务看板

**数据自动从 GitLab 获取，无需手动录入。**

三列看板，映射 GitLab Issue 状态：

| 看板列 | GitLab 状态 | 说明 |
|--------|-------------|------|
| 待办 | opened + 无 assignee 或 label:todo | 待认领/未开始 |
| 进行中 | opened + 有 assignee | 已分配且进行中 |
| 已完成 | closed | 近期关闭的 Issue |

每张卡片自动展示：
- Issue 标题
- 负责人（assignee）
- 所属项目（repo）
- 标签

#### 5.4 工作时间线

**自动从 GitLab Events API 获取。**

按时间倒序展示：
- MR 创建 / 合并
- Issue 打开 / 关闭
- 代码推送
- 评论活动

每条记录自动包含：时间戳 + 执行者 + 事件类型 + 项目

#### 5.5 实时推送（#10）

- **WebSocket 长连接** — 前端打开页面即建立 WS 连接，后端有数据变化时主动推送
- 后端定时轮询信息源（Connect 30s / GitLab 60s），检测到变化后通过 WS 推送
- 页面顶部显示「连接状态」（已连接/重连中）和「最后更新时间」
- 无需手动刷新按钮（但保留作为备用）

## 6. 数据来源

### 6.1 HxA Connect API（Agent 信息 + 在线状态）

```
GET /api/bots          → 所有 Agent 列表 + 在线状态 + profile
GET /api/bots/:name    → 单个 Agent 详情
```

认证：`Authorization: Bearer <agent_token>`

自动获取字段：name, role, bio, tags, online, last_seen_at, team

### 6.2 GitLab API（工作数据）

```
GET /api/v4/groups/:id/issues          → Issue 列表（看板数据）
GET /api/v4/groups/:id/merge_requests  → MR 列表
GET /api/v4/users/:id/events           → 用户活动事件（时间线）
```

认证：`PRIVATE-TOKEN`

自动获取：Issue 状态/assignee/labels, MR 状态/reviewer, 活动事件

### 6.3 SQLite 缓存

本地缓存 API 响应，减少外部请求频率：
- Agent 信息快照（每次轮询更新）
- GitLab 事件记录（增量追加）
- 统计数据缓存

**不是主数据源** — 真实数据永远从 HxA Connect 和 GitLab 获取，SQLite 只做缓存和历史留存。

### 6.4 初始配置

`config/sources.json` — 配置数据源连接信息（仅一次性配置，非数据录入）：

```json
{
  "connect": {
    "hub_url": "https://connect.example.com/hub",
    "org_id": "8375851a-...",
    "agent_token": "bot_..."
  },
  "gitlab": {
    "url": "https://gitlab.example.com",
    "token": "glpat-...",
    "group_id": 13
  },
  "refresh_interval": 30
}
```

## 7. 技术方案

### 7.1 架构

```
┌─────────────┐     ┌──────────────────┐     ┌──────────────┐
│   浏览器     │ ←── │  Node.js 聚合服务 │ ←── │ HxA Connect  │
│  (HTML/JS)  │     │  :3479           │ ←── │ GitLab API   │
└─────────────┘     │  自动轮询+缓存    │     └──────────────┘
                    │  SQLite           │
                    └──────────────────┘
```

### 7.2 技术选型

| 层 | 技术 | 理由 |
|----|------|------|
| 前端 | HTML + CSS + 原生 JS | 轻量、快速、无构建步骤 |
| 后端 | Node.js + Express | 聚合多数据源，提供统一 API |
| 实时通信 | WebSocket（ws 库） | 后端→前端实时推送（#10） |
| 数据库 | SQLite（better-sqlite3） | 轻量缓存，无需额外服务 |
| 部署 | PM2 + Caddy 反向代理 | 与现有基础设施一致 |
| 认证 | Basic Auth（Caddy 层） | 与 dash.example.com 统一（#4） |

### 7.3 API 设计

后端对前端提供的 API（聚合后的数据）：

| 端点 | 说明 |
|------|------|
| `GET /api/team` | 团队成员列表 + 在线状态 + 当前工作概要 |
| `GET /api/team/:name` | 单个 Agent 详情（含 GitLab 活动） |
| `GET /api/board` | 任务看板数据（自动从 GitLab Issue 映射） |
| `GET /api/timeline` | 工作时间线（自动从 GitLab Events 聚合） |

**没有 POST/PUT 端点** — 面板只读，数据全部从信息源自动获取。

### 7.4 端口与路由

- 服务端口：3479
- 路由：`dash.example.com/hxa-dash/` → `localhost:3479`
- 认证：Basic Auth（Caddy 层，user/password）
